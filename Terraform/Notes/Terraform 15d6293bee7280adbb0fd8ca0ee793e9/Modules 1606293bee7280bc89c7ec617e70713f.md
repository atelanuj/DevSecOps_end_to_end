**Introduction to Terraform module**

Terraform module is a set of Terraform configuration files (*.tf) in a directory. The advantage of using modules is `reusability`. You can use the terraform module available in the terraform registry or share the module you created with your team members.

```hcl
module "ec2_instance" {
  source            = "./modules/ec2"
  
  
  ec2_name          = "Terraform_instance"
  ami_id            = "ami-0e35ddab05955cf57"
  sg_name           = "default_sg_ec2"
  ec2_instance_name = "t2.micro"
  ec2_pem_file      = "DevOps"
  ec2_az            = "ap-south-1b"
  enable_port_80    = 80

}
```

- **`*ec2-instance*`** is the name of the module, and as a source its referencing the directory we have created earlier(where we moved all the *.tf files)
- Next, we are referencing all the values of variables that we have defined inside the variables.tf. This is the advantage of using modules, as now **we don’t need to go inside the variables.tf** to modify the value, and we have one single place where we can refer and modify it.
- The values passed from the `module` block will be overridden the default variables inside the module.
- Modules is a template which you need to configure.

## Examples:

ec2 module link = https://registry.terraform.io/modules/terraform-aws-modules/ec2-instance/aws/latest

```hcl
module "ec2-instance" {
  source  = "terraform-aws-modules/ec2-instance/aws"

  key = value
  key1 = value1

```

- refer the [variables.tf](http://variables.tf) file in modules directory to add `key = value`

## What is the difference between resources and modules in Terraform?

A resource in Terraform describes a piece of infrastructure that is going to be created (e.g., a VPC, a subnet, an ec2 instance, etc), whereas a module is a collection of resources that are used together to achieve a reusable use case.

## **Root module**

- The **entry point** for Terraform execution (your main directory where **`terraform apply`** is run).
- Contains configurations (e.g., **`main.tf`**, **`variables.tf`**, **`outputs.tf`**) and can call **child modules**.
- Example:

```hcl
# ./main.tf - root
module "vpc" {  # Calling a child module
  source = "./modules/aws-vpc"
  cidr   = "10.0.0.0/16"
}
```

## **Child module**

- A **self-contained subdirectory** or external module (e.g., from Terraform Registry) called by the root or another module.
- Encapsulates resources (e.g., AWS VPC, EC2) for reusability across projects.
- Example structure:

```
├── main.tf          (Root Module)
└── modules/
    └── aws-vpc/     (Child Module)
        ├── main.tf
        ├── variables.tf
        └── outputs.tf
```

### **Key Differences**

| Root Module | **Child Module** |  |
| --- | --- | --- |
| Top-level configuration | Called by root or other modules |  |
| Defines provider/backend | Reusable, parameterized components |  |
| Can exist without modules | Requires a root module to execute |  |

## **Local module**

- **Definition**:
    
    Modules stored **within your project’s directory** or a **local file path PC**.
    
    - Typically used for organizing code within the same repository.
    - Not shared outside the project.
- **Example Structure**
    
    ```
    my-project/
    ├── main.tf
    └── modules/
        └── network/
            ├── main.tf
            ├── variables.tf
            └── outputs.tf
    ```
    
- **Usage**
    
    ```hcl
    module "network" {
      source = "./modules/network"  # Local path
      cidr   = "10.0.0.0/16"
    }
    ```
    
- **Use Cases**:
    - Project-specific components (e.g., a custom network setup).
    - Code that doesn’t need to be reused across teams/projects.

## **Published module**

- ****Store reusable modules in Git repositories or the Terraform Registry.
- Modules hosted in **external repositories** or registries for broader reuse.
    - Examples: **Terraform Registry, Git (GitHub/GitLab), S3, or private registries.**
    - Designed to be shared across teams, projects, or organizations.

```hcl
# Public Terraform Registry (AWS VPC module)
module "vpc" {
  source  = "terraform-aws-modules/vpc/aws"
  version = "5.0.0"
}

# Git Repository
module "security" {
  source = "git::https://github.com/my-org/terraform-security-module.git"
}

# Private Registry (Terraform Cloud/Enterprise)
module "database" {
  source  = "app.terraform.io/company/database/aws"
  version = "1.2.0"
}
module "vpc" {
  source = "s3::https://s3.amazonaws.com/my-terraform-modules/modules/vpc"
  # Optional: Specify a versioned object (if bucket versioning is enabled)
  # source = "s3::https://s3.amazonaws.com/my-terraform-modules/modules/vpc?version=abcd1234"
  
  # Pass inputs to the module
  cidr_block = "10.0.0.0/16"
}
```

### **Key Notes for storing the modules in S3 bucket**

1. **Path Structure**:
    - Use the full S3 object path (e.g., **`modules/vpc`** for **`s3://my-terraform-modules/modules/vpc`**).
    - Terraform treats the S3 path as a **directory**, so ensure all **`.tf`** files are included.
2. **Versioning**:
    
    Append **`?version=<VERSION_ID>`** to the source URL if the bucket has **versioning enabled**.
    
3. **Authentication**:
    - Terraform uses the **AWS SDK** to access S3, so credentials must be configured (CLI, env vars, or instance roles).
    - Avoid hardcoding credentials in Terraform files.
4. **Cross-Region Access**
    
    If the bucket is in a non-default region, use the regional endpoint:
    
    ```hcl
    source = "s3::https://s3-<REGION>.amazonaws.com/my-terraform-modules/modules/vpc"
    # Example: us-west-2
    source = "s3::https://s3-us-west-2.amazonaws.com/my-terraform-modules/modules/vpc"
    ```

## Calling a module from another module

```
# Root Module (main.tf)
├── main.tf                 # Calls the parent module (e.g., "app")
└── modules/
    ├── app/                # Parent Module
    │   ├── main.tf         # Calls the child module (e.g., "database")
    │   ├── variables.tf
    │   └── outputs.tf
    └── database/           # Child Module
        ├── main.tf         # Defines AWS RDS resources
        ├── variables.tf
        └── outputs.tf
```

1. **Define the Child Module** (e.g., **`modules/database`**)
    
    ```hcl
    # modules/database/main.tf
    resource "aws_db_instance" "default" {
      allocated_storage = var.storage
      engine            = "mysql"
      instance_class    = var.instance_type
    }
    ```
    
2. **Call the Child Module from the Parent Module** (e.g., **`modules/app`**)
    
    ```hcl
    # modules/app/main.tf
    module "database" {
      source          = "../database"  # Path to child module
      storage         = 20             # Pass variable to child
      instance_type   = "db.t2.micro"
    }
    ```
    
3. **Call the Parent Module from the Root Module**
    
    ```hcl
    # root/main.tf
    module "app" {
      source = "./modules/app"  # Path to parent module
    }
    ```
    

### **Key Rules for passing values (ID, ARN, name) from one module to another module**

1. **No Direct Access**:
    
    You **cannot** directly reference a resource like **`aws_vpc.main.id`** outside its module.
    
    ❌ Invalid: **`aws_vpc.main.id`** (resource is hidden inside the module).
    
    ✅ Valid: **`module.network.vpc_id`** (via module output).
    
2. **Outputs Are Required**:
    
    If you need a resource’s attribute (e.g., ARN, ID, name), it **must be explicitly declared as an output** in the child module.
    
3. **Cross-Module Dependencies**
    
    Outputs from one module can be passed as inputs to another module:
    
    ```hcl
    module "database" {
      source   = "./modules/database"
      vpc_id   = module.network.vpc_id  # Output from "network" module (child module)
      subnet_ids = module.network.subnet_ids
    }
    ```
